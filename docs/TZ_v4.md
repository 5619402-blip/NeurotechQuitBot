# NeuroTech Quit — Финальное техническое задание
## Версия v4 FULL. Назначение: разработка MVP Telegram-бота NeuroTech Quit с внешним защищённым плеером.

**Важно:** ручная оплата через чек не является основным сценарием. Telegram Mini App не входит в MVP. Основной формат MVP: Telegram-бот + внешний защищённый web-плеер.

---

## 1. Цель документа

Документ описывает MVP NeuroTech Quit для реализации в Claude Code. ТЗ должно быть достаточно точным, чтобы разработчик не додумывал бизнес-логику, переходы, экраны, кнопки, статусы, оплату, плеер, follow-up, отзывы и админ-функции.

- Все пользовательские тексты и элементы интерфейса пишутся на русском языке.
- Нельзя добавлять функционал, не согласованный в этом ТЗ.
- Нельзя заменять утверждённые названия кнопок без согласования.
- Если в логике есть неопределённость, Claude Code должен остановиться и задать вопрос, а не придумывать решение самостоятельно.

---

## 2. Формат MVP

MVP реализуется как Telegram-бот + внешний защищённый web-плеер.

- **Telegram-бот** отвечает за вход, диагностику, кнопки, статусы, напоминания, follow-up, поддержку-заглушку и маршрутизацию.
- **Внешний web-плеер** отвечает за защищённое воспроизведение аудиопроцедур.
- **Telegram Mini App** не входит в MVP и может быть реализован на втором этапе.
- **Оплата** реализуется как модуль под будущего платёжного провайдера. До подключения провайдера используется тестовая заглушка оплаты.

---

## 3. Главный пользовательский flow

1. `/start`
2. Welcome / первый экран
3. Объясняющий ролик или отзывы
4. Диагностика: экран 1
5. Диагностика: экран 2
6. Проверка мотивации
7. Пользовательское согласие
8. Выбор тарифа
9. Оплата / тестовая заглушка оплаты
10. Успешная оплата
11. Ролик с правилами
12. Подготовка к процедуре
13. Экран предупреждения в плеере
14. Аудиопроцедура
15. Инструкция после процедуры
16. Обязательные вопросы после процедуры
17. Маршрутизация после ответов
18. Напоминания / follow-up / следующая процедура / завершение цикла

| not_smoking | Показать экран результата / рекомендаций. |

## 5. Экраны MVP

### 5.1 Welcome

Назначение: короткое знакомство с NeuroTech Quit и выбор первого действия.

**Текст экрана:**

«Добро пожаловать в NeuroTech Quit.

Это бот сопровождения по авторскому протоколу помощи при отказе от никотина.

Перед началом вы сможете посмотреть короткое объяснение метода, изучить отзывы и пройти диагностику готовности. После диагностики бот покажет подходящий следующий шаг».

- **Смотреть объяснение** → Объясняющий ролик
- **Смотреть отзывы** → Отзывы до диагностики
- **Пройти диагностику** → Диагностика (первый экран)

### 5.2 Объясняющий ролик

Текст: «Посмотрите короткое объяснение, как формируется никотиновая зависимость и как работает NeuroTech Quit».

- **Смотреть** → показать видео/медиа в Telegram
- **Смотреть отзывы** → отзывы
- **Назад** → Welcome
- После просмотра / продолжения: кнопка **«Пройти диагностику»**. Обязательный досмотр до конца не отслеживается.

### 5.3 Отзывы до диагностики

Показываются витринные отзывы: видео, аудио, текст, 2GIS. Отзывы не хранятся в Telegram. Подробная логика хранения описана в разделе «Отзывы».

- **Пройти диагностику** → диагностика
- **Назад** → Welcome

### 5.4 Диагностика

Диагностика состоит из двух экранов. Ответы сохраняются по мере выбора. Если пользователь выходит, незавершённые ответы сохраняются в draft_answers.

| Экран | Вопросы | Кнопки |
|---|---|---|
| Экран 1 | Имя; возраст; что использует: сигареты / стики / вейп / смешанное; стаж курения. | Далее; Назад |
| Экран 2 | Пробовал ли бросать; сколько раз; сколько максимум не курил; мотивация 1–10 кнопками; личное ли это решение. | Далее; Назад |

### 5.5 Проверка мотивации

- Если мотивация 1–3 или ответ «меня уговорили» → **режим созревания**.
- Если мотивация 4+ и решение личное → **пользовательское согласие**.
- Если мотивация 4+ и «Пока не уверен(а)» → **уточняющий экран**.

### 5.6 Режим созревания

Это отдельный экран, не просто end_session.

Текст: «Сейчас ваше внутреннее решение выглядит недостаточно устойчивым для прохождения протокола. NeuroTech Quit работает лучше, когда решение отказаться от никотина является личным и осознанным».

- **Посмотреть отзывы** → отзывы
- **Вернуться позже** → protocol_paused / сессия приостановлена
- **Всё равно продолжить** → пользовательское согласие

В базе: maturity_mode_enabled = true.

### 5.7 Уточняющий экран «Пока не уверен(а)»

Показывается при motivation_level >= 4 и motivation_source = not_sure.

- **Я хочу продолжить** → пользовательское согласие
- **Посмотреть отзывы** → отзывы
- **Вернуться позже** → сессия приостановлена

### 5.8 Пользовательское согласие

Пользовательское согласие обязательно до оплаты. Без принятого согласия нельзя перейти к оплате или процедуре. Полный текст согласия находится в Приложении А.

- **Согласен, продолжить** → выбор тарифа
- **Назад** → предыдущий экран

## 6. Тарифы и оплата

### 6.1 Тарифы

| Сценарий | Что показывать |
|---|---|
| Новый пользователь | Одна процедура — 990 ₽; Полный доступ — 4 900 ₽. |
| Пользователь оплатил 990 ₽ | Оплатить следующую процедуру — 990 ₽; Доплатить до полного доступа — 3 910 ₽. |
| Пользователь оплатил полный доступ | Оплату больше не показывать. |

### 6.2 Тариф 990 ₽ / single_procedure

После оплаты 990 ₽ система создаёт доступ на одну основную процедуру anti_tobacco и одну поддерживающую процедуру alpha. Это не full_access. Следующая основная процедура бесплатно не открывается.

Поля доступа:
- access_type = single_procedure
- paid_main_procedures_count = 1
- used_main_procedures_count = 0
- available_alpha_sessions_count = 1
- used_alpha_sessions_count = 0
- upgrade_available = true
- upgrade_amount = 3910

Альфа-процедура в тарифе 990 ₽ доступна один раз и не меняет current_step_number.

### 6.3 Заглушка оплаты

До подключения платёжного провайдера экран оплаты работает в тестовом режиме.

Текст: «Оплата временно находится в тестовом режиме. Сейчас можно проверить весь путь прохождения без реального списания средств».

- **Тестово открыть доступ** → создаётся payment со статусом paid, открывается выбранный доступ, пользователь попадает на экран «Оплата прошла успешно»
- **Назад к выбору тарифа**

После подключения провайдера заглушка заменяется на реальную оплату, но механизм открытия доступа остаётся тем же.

### 6.4 Успешная оплата

Текст: «Оплата прошла успешно. Доступ открыт. Перед запуском процедуры посмотрите короткие правила прохождения. Это поможет пройти её правильно и без лишних ошибок».

- **Перейти к правилам** → ролик с правилами

### 6.5 Неуспешная оплата

Показывается, если платёж не прошёл или был отменён.

- **Повторить оплату**
- **Выбрать другой тариф**
- **Назад**

## 7. Правила и подготовка к процедуре

### 7.1 Ролик с правилами

Текст: «Перед процедурой посмотрите короткие правила. Это важно для корректного прохождения».

- **Смотреть**
- **Перейти к подготовке**
- **Назад** → Мой доступ (возврат на экран успешной оплаты не используется, чтобы не создавать путаницу)

### 7.2 Подготовка к процедуре: первая процедура

1. Последний контакт с никотином, предусмотренный протоколом, завершён непосредственно перед процедурой, не за час и не за два, а перед началом сессии, как прощальный ритуал.
2. Пользователь собрал и выбросил все курительные принадлежности и триггеры в доме: сигареты, стики, вейпы, нагреватели, зажигалки, спички и всё, что может вернуть старый сценарий.
3. В доме не осталось курительных триггеров.
4. Пользователь в трезвом и спокойном состоянии.
5. Пользователь выбрал спокойное место, желательно тёмную комнату, где его никто не будет отвлекать.
6. Телефон выключен или убран, уведомления и внешние раздражители отключены.
7. Пользователь сидит удобно, не лежит.
8. Наушники надеты.
9. Пользователь готов пройти процедуру полностью и без остановок.

- **Да, я готов** → плеер
- **Мне нужно подготовиться** → helper-text на этом же этапе (см. 7.4)
- **Назад** → зависит от источника: если пользователь пришёл из «Ролик с правилами» (7.1) — назад к ролику с правилами; если из «Мой доступ», напоминания или иного источника — в «Мой доступ». Для реализации хранить `previous_screen` в состоянии сессии. Если источник неизвестен — в «Мой доступ».

### 7.3 Подготовка к процедуре: следующие процедуры

1. Спокойное место
2. Телефон убран
3. Пользователь сидит удобно
4. Наушники надеты
5. Готовность пройти процедуру полностью и без остановок

- **Да, я готов** → плеер
- **Мне нужно подготовиться** → helper-text (см. 7.4)
- **Назад** → Мой доступ

### 7.4 Кнопка «Мне нужно подготовиться»

Кнопка не создаёт отдельный flow. При нажатии процедура не запускается, procedure_session не создаётся, доступ не списывается. Пользователь остаётся на этапе подготовки.

| Контекст | Текст | Кнопки |
|---|---|---|
| Первая процедура | Подготовьтесь спокойно и возвращайтесь, когда будете готовы пройти процедуру полностью без остановок. Перед первой процедурой важно завершить последний контакт с никотином непосредственно перед сессией и убрать все курительные триггеры из дома. | Я готов; Вернуться в Мой доступ |
| Следующие процедуры | Подготовьтесь спокойно и возвращайтесь, когда будете готовы пройти процедуру полностью без остановок. | Я готов; Вернуться в Мой доступ |

## 8. Внешний защищённый плеер

- Плеер не покупается отдельно, а разрабатывается как часть проекта.
- Аудио не хранится в Telegram и не отправляется пользователю.
- Аудио хранится во внешнем приватном хранилище, доступном из РФ и через VPN.
- Размер аудиофайлов ориентировочно 100–200 МБ каждый.
- Все аудио должны сохранять stereo L/R. Нельзя переводить в mono и нельзя ломать бинауральные ритмы.
- Доступ только через временную защищённую ссылку / токен.
- **TTL player token: 2 часа.** Если токен истёк, бот создаёт новый при повторном нажатии.

### 8.1 Ограничения плеера

- Нет паузы
- Нет перемотки
- Нет таймлайна
- Нет скачивания
- Нет прямой ссылки на аудио
- Во время процедуры единственная кнопка: **«Экстренно выйти»**

### 8.2 Экран перед запуском

Заголовок: «Важное правило прохождения».

Текст: «Процедуру необходимо проходить полностью и без остановок. Нельзя ставить аудио на паузу, выходить из процедуры или отвлекаться. Последовательность аудио и инструкций имеет значение для корректной работы метода. Если процедура была прервана, остановлена или пользователь отвлекался, NeuroTech не может гарантировать корректность результата этой сессии. В таком случае процедуру рекомендуется пройти заново с самого начала».

- **Начать процедуру**
- **Вернуться назад**

### 8.3 Экстренный выход

При нажатии «Экстренно выйти» процедура не засчитывается: session_status = interrupted, is_counted_as_completed = false.

Текст: «Процедура была прервана. Для корректного результата её нужно будет пройти заново с самого начала».

- **Начать заново позже** → Мой доступ

### 8.4 Механизм «плеер → backend → бот»

1. После полного завершения аудио плеер отправляет **server callback** на backend с procedure_session_id и событием завершения.
2. Backend меняет status на completed и сохраняет completed_at.
3. После этого бот показывает пользователю экран «После процедуры / фиксация состояния».

Механизм не зависит от Telegram WebApp API и должен работать через обычный браузер / Telegram WebView.

**Таймаут callback:**

Если `procedure_session.session_status = started` и server callback не поступил в течение `audio_files.duration_seconds + 30 минут`, backend автоматически помечает сессию: `session_status = abandoned`, `is_counted_as_completed = false`.

При следующем входе пользователя в бот (через `/start` или «Мой доступ») показывается сообщение:

«Процедура не была завершена корректно. Для корректного результата начните её заново».

- **Начать заново** → Подготовка к процедуре (7.3)
- **Мой доступ**

## 9. Порядок процедур

- **Базовый минимум:** Анти-табак → Быстрый рычаг → Анти-табак.
- Если нужно продолжать, далее чередование: Быстрый рычаг → Анти-табак → Быстрый рычаг → Анти-табак и далее по состоянию.
- Пользователь не выбирает порядок основных процедур сам до завершения базового протокола.
- **Альфа-процедура** является дополнительной поддерживающей процедурой и не меняет current_step_number.
- Альфа может быть доступна между основными процедурами или по состоянию пользователя.

### 9.1 Альфа-процедура

**Вход:** из «Мой доступ», из рекомендаций, после follow-up, после тревоги / тяги.

Перед Альфа используется сокращённая подготовка (см. 7.3). Плеер тот же: без паузы, без перемотки, без скачивания. После Альфа обязательные вопросы сохраняются, но current_step_number основного протокола не меняется.

**Лимиты Альфа-процедуры:**

| access_type | available_alpha_sessions_count | Комментарий |
|---|---|---|
| single_procedure | 1 | Одна Альфа включена в тариф |
| full_access | NULL (безлимитно) | Альфа доступна без ограничений |

Альфа не открывает следующую основную процедуру и не сбивает чередование anti_tobacco / quick_lever.

### 9.2 Алгоритм next_procedure_type

После завершения каждой основной процедуры backend выполняет три шага:
1. `protocol_progress.current_step_number` увеличивается на 1.
2. `next_procedure_type` вычисляется по правилу: если `current_step_number` **чётный** → `anti_tobacco`; если **нечётный** → `quick_lever`.
3. `users.next_procedure_id` обновляется ссылкой на соответствующую запись в таблице `procedures`.

**Таблица маппинга:**

| current_step_number (завершено) | Порядковый № следующей процедуры | next_procedure_type |
|---|---|---|
| 0 | 1 | anti_tobacco |
| 1 | 2 | quick_lever |
| 2 | 3 | anti_tobacco |
| 3 | 4 | quick_lever |
| 4 | 5 | anti_tobacco |

Начальное значение `current_step_number = 0` (ни одной основной процедуры не пройдено). Первая процедура — всегда `anti_tobacco`.

**Примечание по чётности:** `current_step_number` — счётчик завершённых процедур, начинается с 0. Порядковый номер следующей процедуры = `current_step_number + 1`. Поскольку 0 чётный → первая процедура `anti_tobacco`. Порядковые номера процедур в протоколе имеют обратную чётность относительно `current_step_number`; для реализации использовать именно значение `current_step_number`.

**Альфа-процедура не меняет `current_step_number`.** После Альфа: счётчик остаётся прежним, `next_procedure_type` не пересчитывается, чередование `anti_tobacco / quick_lever` не нарушается.

## 10. «Мой доступ», главное меню и продолжение

### 10.1 «Мой доступ»

«Мой доступ» — динамический экран. Он строится по данным пользователя в базе:
- Тариф
- Статус доступа
- Текущий этап
- Следующая рекомендованная процедура
- Количество завершённых процедур
- Наличие незавершённой процедуры
- Доступность Альфа-процедуры
- Кнопка продолжения

| Условие | Кнопки |
|---|---|
| full_access | Продолжить протокол; Альфа-процедура; Поддержка; Главное меню |
| single_procedure | Пройти оплаченную процедуру; Оплатить следующую процедуру — 990 ₽; Доплатить до полного доступа — 3 910 ₽; Альфа-процедура, если доступна; Поддержка; Главное меню |

**Важно:** Отдельный экран «Мой прогресс» не создаётся. Прогресс показывается внутри «Мой доступ». Если в интерфейсе нужна кнопка «Мой прогресс», она ведёт на «Мой доступ» / раскрывает тот же блок прогресса.

### 10.2 Кнопка «Продолжить протокол»

Кнопка не ведёт на статичный экран. Она запускает умную проверку статуса пользователя.

| Условие | Переход |
|---|---|
| Есть незавершённая процедура | Экран: процедура была прервана / начать заново |
| Следующая процедура доступна | Подготовка к процедуре |
| Нужна оплата следующей процедуры | Выбор тарифа / оплата |
| user_status = not_smoking | Экран результата / рекомендаций (13.2) |

`user_status = not_smoking` устанавливается через Q5 = «Я уже не курю» в разделе 11.2, через ответ «Я уже не курю» в next_day_followup (12.1) или через кнопку «Я уже бросил курить» в напоминании (12.2).

### 10.3 Главное меню

Главное меню — это **не Welcome**. Оно используется для пользователей, которые уже находятся внутри сценария.

Кнопки: **Мой доступ; Отзывы; Поддержка; О проекте**

## 11. После процедуры и обязательные вопросы

### 11.1 Экран после завершения процедуры

Показывается после завершения плеера и получения callback от backend.

Смысл: поздравить, дать инструкцию наблюдать за состоянием, воздержаться от никотина, не приближаться к триггерам, при тяге вспомнить заключительную эмоциональную часть процедуры / нейродинамической медитации.

- **Далее** → обязательные вопросы

### 11.2 Обязательные вопросы после процедуры

- Нельзя пропустить вопросы.
- Ответы сохраняются автоматически после каждого ответа в `post_procedure_answers.answers_json`.
- Без завершения всех обязательных вопросов нельзя перейти к следующей процедуре, оплате следующей процедуры, follow-up или другому этапу.
- После всех ответов появляется кнопка **«Продолжить»**, после которой применяется маршрутизация из раздела 11.3.

**Q1–Q4 — аналитика** (сохраняются в answers_json, не меняют user_status напрямую):

**Q1. «Как вы сейчас себя чувствуете?»**
Кнопки: «Спокойно»; «Легче»; «Напряжённо»; «Непривычно»; «Пока не понимаю»

**Q2. «Есть ли сейчас желание курить?»**
Кнопки: «Нет»; «Слабое»; «Среднее»; «Сильное»

**Q3. «Было ли желание прервать процедуру или сопротивление во время прохождения?»**
Кнопки: «Нет»; «Немного»; «Да, было заметно»; «Сложно сказать»

**Q4. «Как сейчас воспринимается мысль о никотине?»**
Кнопки: «Не тянет»; «Есть отстранение»; «Есть сомнение»; «Тянет как раньше»; «Пока не понимаю»

**Q5 — основной routing-вопрос** (меняет user_status и определяет переход по разделу 11.3):

**Q5. «Нужна ли вам следующая процедура по протоколу?»**
Кнопки: «Да, хочу продолжить»; «Пока не знаю»; «Я уже не курю»; «Не хочу продолжать»

**Q6 — вспомогательный вопрос** (показывается всем пользователям после Q5; ответ учитывается при маршрутизации согласно разделу 11.3):

**Q6. «Нужна ли вам помощь или поддержка?»**
Кнопки: «Нет»; «Да, нужна поддержка»; «Хочу пройти Альфа-процедуру»

### 11.3 Маршрутизация после обязательных вопросов

Routing определяется ответами Q5 и Q6. Q5 определяет изменение `user_status` и имеет приоритет. Q6 уточняет экран назначения: при Q5 = «Пока не знаю» — Q6 полностью определяет переход; при других значениях Q5 — Q6 = «Да, нужна поддержка» делает поддержку доступной из экрана назначения (не меняет сам переход).

| Q5 | Q6 | user_status после | Действие |
|---|---|---|---|
| Да, хочу продолжить | любой | waiting_next_procedure | Следующая основная процедура (алгоритм 9.2), если доступна. Если нужна оплата — экран оплаты / апгрейда. |
| Пока не знаю | Хочу пройти Альфа-процедуру | без изменения | → Альфа-процедура (сокращённая подготовка 7.3), если доступна. Иначе → Мой доступ. |
| Пока не знаю | Да, нужна поддержка | без изменения | → Экран «Поддержка» (14). Напоминание создаётся по схеме 12.2. |
| Пока не знаю | Нет | без изменения | → Мой доступ. Напоминание создаётся по схеме 12.2. |
| Я уже не курю | любой | not_smoking | Создать seven_day_followup. → Экран результата / рекомендаций (13.2). |
| Не хочу продолжать | любой | protocol_paused | → Экран «Сессия приостановлена» (13.1). |

**Дополнительное правило (Q2 = «Сильное»):** если Q5 = «Да, хочу продолжить» и Q2 = «Сильное» — перед переходом к следующей основной процедуре показать «Мой доступ» с рекомендацией Альфа-процедуры, если доступна.

**Срыв:** обнаруживается не через Q1–Q6 (процедура только что завершена), а через next_day_followup (12.1) — вопрос «Курили ли вы после процедуры?». При ответе о срыве: мягкий экран «Это не отменяет прогресс» с кнопками «Продолжить протокол» и «Поддержка».

## 12. Follow-up и напоминания

### 12.1 Follow-up на следующий день

Отправляется через **24 часа от completed_at** процедуры. В таблице next_day_followups обязательно хранить scheduled_at и sent_at.

| Экран | Вопросы / кнопки |
|---|---|
| Состояние и курение | Курили ли вы после процедуры? Нет / Один раз / Несколько раз / Да, вернулся(ась).<br>Есть ли сейчас желание курить? Нет / Слабое / Среднее / Сильное.<br>Стало ли легче воздерживаться? Да, заметно / Немного / Пока не понимаю / Нет. |
| Триггеры и продолжение | Где тяга появлялась сильнее всего? Утром / После еды / С кофе / В стрессе / За рулём / Когда рядом курили / Тяги почти не было.<br>Что заметили? Больше спокойствия / Меньше тяги / Меньше автоматизма / Появилось сопротивление / Пока ничего / Стало тревожнее.<br>Хотите продолжить? Да, продолжить / Я уже не курю / Не хочу продолжать. |

| Ответ (wants_to_continue) | user_status | Действие |
|---|---|---|
| Да, продолжить | waiting_next_procedure | «Мой доступ» / следующая процедура по алгоритму 9.2. |
| Я уже не курю | not_smoking | Показать короткое поздравление, создать seven_day_followup. |
| Не хочу продолжать | protocol_paused | «Мой доступ» с блоком о паузе (раздел 4) или «Сессия приостановлена» (13.1). |

Если `current_craving = «Сильное»` при любом варианте ответа — дополнительно предложить «Пройти Альфа-процедуру», если доступна по тарифу.

**Pre-send проверки перед отправкой next_day_followup:**
- `user_status ≠ not_smoking`
- `user_status ≠ protocol_paused`
- Соответствующая `procedure_session.session_status = completed` (follow-up актуален)
- Пользователь не заблокировал бота (ошибка 403 при отправке не возникала)

Если хотя бы одна проверка не прошла — `next_day_followups.sent_at` не заполняется, запись помечается `cancelled`.

**Когда устанавливается `user_status = followup_pending`:**

`user_status = followup_pending` устанавливается сразу после того, как backend запланировал `next_day_followup` (scheduled_at = completed_at + 24ч) при Q5 = «Пока не знаю». При Q5 = «Да, хочу продолжить» основное состояние — `waiting_next_procedure`; при Q5 = «Не хочу продолжать» — `protocol_paused`; при Q5 = «Я уже не курю» — `not_smoking` (seven_day_followup запускается в фоне независимо от user_status).

`user_status = followup_pending` снимается:
- при получении ответа на next_day_followup — статус обновляется согласно ответу пользователя;
- если follow-up отменён pre-send проверками — статус возвращается к актуальному состоянию: `waiting_next_procedure`, `protocol_paused`, `not_smoking` или `protocol_active`.

### 12.2 Напоминания о следующей процедуре

После каждой завершённой основной процедуры backend создаёт **серию из максимум 3 напоминаний** о следующей положенной процедуре. Вся серия хранится в одной строке таблицы `reminders`.

**Интервалы:**
- 1-е напоминание: через **48 часов** от `completed_at` основной процедуры
- 2-е напоминание: через **24 часа** после 1-го, если пользователь не нажал ни одну кнопку
- 3-е напоминание: через **24 часа** после 2-го, если пользователь не нажал ни одну кнопку

**Логика серии в таблице reminders:**
1. Создаётся запись: `scheduled_at = completed_at + 48ч`, `reminder_count = 0`, `reminder_status = scheduled`
2. В `scheduled_at`: pre-send проверки → отправить → `sent_at = now`, `reminder_count++`, `reminder_status = sent`
3. Через 24ч: если `user_response = null` и `reminder_count < 3` → `scheduled_at = sent_at + 24ч`, `reminder_status = scheduled`, повторить шаг 2
4. Если `user_response = null` и `reminder_count = 3` → `reminder_status = expired`, `user_status = protocol_paused`

Поле `scheduled_at` обновляется после каждой отправки и содержит время следующей попытки. Новые поля в таблице `reminders` не добавляются.

**Pre-send проверки перед каждой отправкой:**
- Не прошёл ли пользователь следующую процедуру (`procedure_sessions` с `session_status = completed`)
- Не установлен ли `user_status = protocol_paused`
- Не установлен ли `user_status = not_smoking`
- `reminder_count < 3`

Если хотя бы одна проверка не прошла — `reminder_status = cancelled`, отправка не производится.

**Значения поля `user_response`:**

Поле устанавливается при нажатии пользователем любой кнопки напоминания:

| Нажатая кнопка | user_response |
|---|---|
| «Пройти следующую процедуру» | continue_procedure |
| «Я уже бросил курить» | already_not_smoking |
| «Не хочу продолжать» | do_not_continue |

При любом ответе: `reminder_status = completed`, новые напоминания по этой серии не создаются.

**После 3 безответных напоминаний:**
- `reminder_status = expired`
- Новые напоминания по этой следующей процедуре не создаются
- `user_status = protocol_paused`
- При следующем `/start` пользователь попадает в «Мой доступ» с информацией о приостановленном протоколе

**Кнопки напоминания:**
- **Пройти следующую процедуру** → подготовка к следующей положенной процедуре или оплата, если доступ нужен
- **Я уже бросил курить** → показать короткое поздравление, создать seven_day_followup, `user_status = not_smoking`
- **Не хочу продолжать** → показать экран «Сессия приостановлена», `user_status = protocol_paused`

### 12.3 Follow-up через 7 дней

Создаётся, если пользователь ответил «Я уже не курю» / «Да, не курю».

trigger_source может быть: after_procedure_not_smoking, next_day_not_smoking, final_check_not_smoking, manual_admin.

**Вопросы:**
- Курили ли за последние 7 дней
- Есть ли желание курить
- Что изменилось
- Нужна ли поддержка дальше

Если пользователь не курит и тяги нет/она слабая, бот показывает красивое поздравление и обязательно предлагает оставить отзыв.

**Кнопки:** Оставить отзыв; Отправить другу; Пройти Альфа-процедуру; Завершить цикл; Главное меню

**Исход «Всё ещё курю» / «Тяга сохраняется»:**

Если пользователь ответил, что курил, вернулся к курению или тяга сильная, бот показывает экран «Результат требует продолжения»:

«Спасибо за честные ответы. Если тяга сохраняется или вы снова курили, это не означает, что весь прогресс потерян. Никотиновая зависимость может требовать нескольких этапов работы. Вы можете продолжить протокол, пройти следующую процедуру или вернуться к поддерживающей Альфа-процедуре, если она доступна по вашему тарифу.»

- **Продолжить протокол** → умная маршрутизация из раздела 10.2
- **Пройти Альфа-процедуру** → проверка доступа к alpha; если доступно — подготовка (7.3); если нет — «Мой доступ»
- **Мой доступ**
- **Поддержка** → экран «Поддержка» (14)

Поле `result_status` в таблице `seven_day_followups`:
- `not_smoking` — пользователь не курит, тяга нет/слабая
- `reduced_smoking` — снизил курение, но тяга сохраняется
- `relapsed` — вернулся к курению
- `unknown` — ответы неопределённые

## 13. Завершение, результат, end_session

### 13.1 Сессия приостановлена / end_session

Показывается в момент, когда пользователь впервые выбирает «Вернуться позже», «Не хочу продолжать» или аналогичный вариант паузы. При последующих входах через `/start` с `user_status = protocol_paused` пользователь попадает на «Мой доступ» с блоком о паузе (см. раздел 4), а не на этот экран.

Текст: «Хорошо. Вы можете вернуться к NeuroTech Quit позже. Когда будете готовы, просто откройте бот снова — мы продолжим с того места, где вы остановились».

- **Вернуться в главное меню**
- **Пройти диагностику заново**

В базе: user_status = protocol_paused, current_screen, pause_reason, paused_at.

**Механика кнопки «Пройти диагностику заново»:**

- Создаётся новая запись в таблице `diagnostics`.
- `user_status = diagnostic_started`.
- Предыдущая запись `diagnostics` не удаляется; сохраняется по `created_at` или помечается флагом (реализация на усмотрение разработчика).
- `access_rights`, `payments`, `procedure_sessions`, `protocol_progress` **не сбрасываются**.
- Новая диагностика не обнуляет оплаченный доступ и историю процедур пользователя.

### 13.2 Экран результата / рекомендаций

Показывается для user_status = not_smoking.

Текст: «Сейчас ваше состояние выглядит стабильнее. Постарайтесь избегать проверки себя "на одну сигарету", не возвращаться к старым триггерам и наблюдать за изменениями спокойно».

- **Альфа-процедура**
- **Оставить отзыв**
- **Главное меню**

### 13.3 Кнопка «Завершить цикл»

Кнопка не должна открывать бесплатный доступ к следующим процедурам без проверки тарифа.

| Тип доступа | Экран / кнопки |
|---|---|
| full_access | Экран «Цикл завершён». Кнопки: Мой доступ; Оставить отзыв; Отправить другу; Главное меню. |
| single_procedure | Экран «Текущий этап завершён». Кнопки: Оплатить следующую процедуру — 990 ₽; Доплатить до полного доступа — 3 910 ₽; Оставить отзыв; Главное меню. |

- **Главное меню**

## 16. Админ-функции и подарочный доступ

### 16.1 Где хранить Telegram ID администратора

Для MVP **не создавать таблицу admins**. Использовать ENV-переменную:

```
ADMIN_TELEGRAM_IDS=123456789,987654321
```

При админ-командах бот проверяет Telegram ID. Если ID нет в ENV, доступ запрещён.

### 16.2 Подарочный доступ

**Команда:** `/gift` или кнопка в админ-меню «Создать подарочный доступ». Доступно только администраторам из ADMIN_TELEGRAM_IDS.

- **Типы доступа (поле `gift_access_type`):** gift_single_procedure, gift_full_access, gift_alpha, gift_test_access.
- **Сроки:** 24 часа, 3 дня, 7 дней.
- Подарочная ссылка одноразовая, временная, активируется первым пользователем.
- Диагностика и пользовательское согласие всё равно обязательны, оплата не требуется.

**Маппинг gift_access_type → access_type после активации:**

| gift_access_type | access_type в access_rights | Описание |
|---|---|---|
| gift_single_procedure | single_procedure | Одна основная процедура |
| gift_full_access | full_access | Полный доступ |
| gift_alpha | none (отдельный флаг) | Только Альфа-процедура; основные процедуры не открываются |
| gift_test_access | none (по правилам админа) | Объём определяется администратором при создании токена |

`gift_access_type` хранится в таблице `gift_access_tokens` как отдельное поле — не в основном ENUM `access_type`. Это позволяет не ломать логику основного доступа.

### 16.3 Активация подарочного доступа пользователем

**Формат ссылки:** `https://t.me/NeuroTechQuitBot?start=gift_XXXX`

При `/start` с параметром `gift_XXXX` бот выполняет следующие шаги:

1. Извлекает токен из параметра.
2. Ищет его в таблице `gift_access_tokens`.
3. Проверяет: `status = active` и `expires_at > now()` и `activated_by_user_id IS NULL`.
4. **Токен недействителен или истёк** — показать экран: «Подарочная ссылка недействительна или истекла. Если у вас есть другая ссылка, попробуйте её». Кнопки: «Главное меню»; «Пройти диагностику».
5. **Токен действителен** — привязать к пользователю:
   - `gift_access_tokens.activated_by_user_id = telegram_id`
   - `gift_access_tokens.activated_at = now()`
   - `gift_access_tokens.status = used`
   - Создать `access_rights` согласно маппингу `gift_access_type` → `access_type` из раздела 16.2.

**Маршрутизация после успешной активации:**

| Состояние пользователя | Следующий шаг |
|---|---|
| Диагностика не пройдена | → Диагностика (5.4) |
| Диагностика пройдена, согласие не принято | → Пользовательское согласие (5.8) |
| Диагностика и согласие пройдены | → «Мой доступ» (10.1) |

Диагностика и пользовательское согласие обязательны даже при подарочном доступе.

## 18.1 ENUM / статусы

**access_type:** none, single_procedure, full_access

**access_status:** inactive, active, expired, blocked

**payment_status:** pending, paid, failed, cancelled, refunded

**session_status:** created, started, completed, interrupted, abandoned, expired

**reminder_status:** scheduled, sent, cancelled, completed, expired

**reminder_type:** next_procedure_48h, next_day_followup, seven_day_followup, manual_admin

**support_status:** new, in_progress, resolved, closed

**moderation_status:** pending, approved, rejected

**user_status:**
```
new

diagnostic_completed
low_motivation
not_sure_clarification
consent_accepted
tariff_selected
payment_pending
paid_single

procedure_in_progress
procedure_interrupted
procedure_completed
waiting_next_procedure
followup_pending
protocol_active
protocol_paused
protocol_completed
not_smoking
stopped_by_user
```

**gift_access_tokens.status:** active, used, expired, revoked

- `active` — ссылка создана и может быть активирована
- `used` — ссылка уже активирована пользователем
- `expired` — срок действия ссылки истёк (`expires_at` прошёл)
- `revoked` — ссылка вручную отключена администратором

Правила переходов:
- Новая ссылка создаётся со статусом `active`
- После успешной активации: `active` → `used`
- Если `expires_at < now()`: `active` → `expired`
- Если администратор отменил: любой → `revoked`
- Активировать можно только ссылку со статусом `active` и `expires_at > now()`

**gift_access_type:** gift_single_procedure, gift_full_access, gift_alpha, gift_test_access

**next_day_followups.wants_to_continue:** continue_protocol, already_not_smoking, do_not_continue, needs_support, unknown

**seven_day_followups.result_status:** not_smoking, reduced_smoking, relapsed, unknown

## 19. Проверка целостности перед кодингом

- [ ] Нельзя показывать следующую процедуру без completed у текущей procedure_session.
- [ ] Нельзя считать процедуру завершённой без server callback от плеера.
- [ ] Нельзя показывать оплату пользователю с full_access.
- [ ] Нельзя открывать следующую основную процедуру бесплатно пользователю single_procedure без оплаченной процедуры.
- [ ] Нельзя смешивать reviews и user_reviews.
- [ ] Нельзя ломать порядок основного протокола.
- [ ] Нельзя давать прямую ссылку на аудиофайл.
- [ ] Нельзя хранить аудио в Telegram.
- [ ] Нельзя запускать процедуру без принятого пользовательского согласия.
- [ ] Нельзя запускать процедуру без оплаты / доступа / подарочного доступа.
- [ ] Нельзя возвращать пользователя на Welcome, если у него уже есть активный прогресс.
